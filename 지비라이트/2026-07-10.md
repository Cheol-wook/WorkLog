## 2026-07-10 업무 기록

### 프로젝트

- 지비라이트 제조표준서 관리 기능

### 작업 목표

- 제조표준서 조회 이후 저장, 확정, 개정 시작까지 이어지는 화면 상태와 데이터 흐름을 안정화했다.
- 문서번호 관리 방식을 정리하고, 사용자가 잘못된 순서로 저장을 실행하는 경우를 차단할 수 있도록 보완했다.

### 수행 작업

- 전일 작성한 상세 조회 바인딩 로직을 리뷰하고, 목록 선택 후 상세 화면에 데이터가 표시되는 흐름을 점검했다.
- 확정 처리 로직을 저장 로직과 분리해, 확정에 필요한 최소 정보만 전달하는 전용 처리 흐름으로 정리했다.
- 개정 시작 프로세스를 설계하고, 확정된 데이터를 다시 작성 가능한 임시 상태로 복사하는 구조를 구성했다.
- 개정 요청 팝업과 제조표준서 화면 사이에 전달해야 하는 값과 호출 순서를 정리했다.
- 기존 전역 변수 중심의 문서번호 관리 방식을 숨김 컨트롤 기반으로 바꾸는 방향을 잡았다.
- 목록 셀 클릭 시 선택한 문서번호를 내부 명령 이벤트로 전달하도록 처리 방식을 정리했다.
- 신규 작성 초기화 없이 저장을 시도하는 경우를 막기 위한 저장 전 가드 조건을 설계했다.

### 기술 스택 및 도구

- C#
- Windows Forms
- SQL Server
- T-SQL
- 저장 프로시저
- DataTable / DataSet
- 문서 상태 기반 업무 흐름 설계

### 문제 해결

- 확정 처리가 기존 저장 로직에 묶여 있어 불필요한 데이터 전달과 호출 오류가 발생할 수 있는 구조를 발견했다.
- 문서번호가 전역 변수와 화면 상태에 나뉘어 관리되면서 저장, 확정, 개정 시점에 혼선이 생길 가능성을 확인했다.
- 개정 시작 시 부모 데이터와 상세 데이터의 복사 순서가 중요하다는 점을 확인하고, 부모 데이터를 먼저 준비하는 방향으로 정리했다.
- 신규 버튼을 누르지 않은 상태에서 저장하면 필수 그리드 데이터가 준비되지 않아 오류가 날 수 있어 사전 차단 조건을 추가하기로 했다.

### 결과

- 조회, 상세 바인딩, 확정, 개정 시작으로 이어지는 제조표준서 주요 흐름이 더 명확해졌다.
- 문서번호 관리 기준이 화면의 숨김 컨트롤로 통일되어 이후 저장과 확정 호출부를 단순화할 수 있게 됐다.
- 확정과 개정 시작 기능을 각각 독립적으로 테스트할 수 있는 구조가 마련됐다.
- 다음 단계에서 문서 삭제와 개정 폐기 기능을 추가할 수 있는 기반을 정리했다.

### 다음 작업

- 확정 및 개정 시작 호출부의 문서번호 참조를 새 기준에 맞춰 정리한다.
- 개정 시작 후 목록 재조회와 상세 재바인딩을 실제 화면에서 테스트한다.
- 문서 삭제와 개정 폐기 기능의 프로세스와 예외 조건을 설계한다.
