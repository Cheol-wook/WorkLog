# 2026-07-08 개발 기록

## 지비라이트 제조표준서 기능 구현

### 작업 목표

- 제조표준서의 저장, 확정, 조회 흐름을 화면 구조와 데이터 생명주기에 맞춰 정리한다.
- 작성 중 데이터, 확정 데이터, 개정 중 데이터를 한 화면에서 일관되게 조회할 수 있는 기준을 설계한다.

### 수행 작업

- 제조표준서의 신규 작성, 작성 중 수정, 확정 처리 흐름을 다시 점검했다.
- 여러 상세 영역을 하나의 부모 정보와 연결해 저장하고 확정하는 구조를 검토했다.
- 확정 시 임시 데이터가 현재 데이터와 이력 데이터로 반영되는 순서를 정리했다.
- 확정 처리에서 상세 데이터와 부모 데이터의 삭제 동기화가 필요한 범위를 확인했다.
- 코팅 영역의 저장 구조 변경 사항을 반영해 초기 저장 및 수정 흐름을 재검토했다.
- 조회 화면을 왼쪽 목록 조회와 오른쪽 상세 조회로 분리하는 방향을 정리했다.
- 작성 중 또는 개정 중인 데이터가 있으면 확정 데이터보다 우선 표시되도록 조회 기준을 설계했다.
- 상세 조회에서 기본정보, 담당자 참관 여부, 파트 정보, 수지, 코팅, 전사 데이터를 순서대로 불러오는 구조를 검토했다.
- 문서 상태값을 작성중, 개정중, 확정 상태로 구분해 화면에 활용하는 방향을 정리했다.
- 문서관리 테이블과 제조표준서 데이터의 연결 키를 확인하고, 조회 쿼리의 조인 기준을 검증했다.

### 기술 스택 및 키워드

- C# / WinForms
- SQL Server / T-SQL
- Table-Valued Parameter
- MERGE 기반 저장 처리
- TEMP / MAIN / REV 데이터 흐름
- 트랜잭션 기반 확정 처리
- 마스터-디테일 조회 화면
- 문서 상태 기반 데이터 우선순위 처리

### 문제 해결

- 작성 중 데이터와 확정 데이터가 동시에 존재할 때 어떤 데이터를 조회해야 하는지 기준을 정리했다.
- 화면의 왼쪽 목록에는 문서 상태와 무관하게 최신 작업본이 보여야 하므로, 임시 데이터가 있으면 해당 데이터를 우선 사용하도록 설계했다.
- 상세 조회에서는 같은 문서번호를 기준으로 기본정보와 하위 상세 데이터를 함께 가져오도록 연결 방식을 정리했다.
- 코팅 상세 영역은 저장 순서와 조회 표시 순서가 달라도 업무적으로 문제가 없도록 정렬 기준을 분리했다.
- 문서 상태 조회는 제조표준서 본문 데이터가 아니라 문서관리 데이터와의 연결을 통해 확인하도록 정리했다.

### 결과

- 신규 저장, 작성 중 수정, 신규 확정에 대한 전체 흐름을 점검했다.
- 확정 처리에 필요한 기본정보, 담당자 참관 여부, 파트, 수지, 코팅, 전사 영역의 처리 방향을 정리했다.
- 목록 조회 쿼리와 상세 조회 쿼리의 기본 구조를 잡았다.
- 상세 조회 결과를 화면의 기본정보 영역과 여러 그리드에 나누어 바인딩할 수 있는 데이터 구조를 정리했다.

- 조회 결과의 컬럼명과 화면 DataTable 컬럼명을 최종 대조한다.
