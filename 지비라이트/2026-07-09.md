# 2026-07-09 업무 기록

## 프로젝트

- 지비라이트 제조표준서 화면 개발

## 작업 목표

- 제조표준서 조회 화면에서 목록 조회와 상세 조회를 분리한다.
- 선택한 문서의 상세 정보를 화면 컨트롤과 여러 그리드에 안정적으로 바인딩하는 기준을 정한다.
- 작성, 저장, 확정으로 이어지는 화면 상태값 관리 기준을 정리한다.

## 수행 작업

- 왼쪽 목록 그리드 조회 흐름을 검토했다.
- 작성 중인 데이터가 있으면 확정 데이터보다 우선 표시되도록 조회 우선순위를 확인했다.
- 목록 그리드의 표시 컬럼과 실제 조회 결과 컬럼 간 매핑 방식을 정리했다.
- 목록 선택 시 오른쪽 상세 화면을 불러오는 이벤트 흐름을 설계했다.
- 상세 조회 결과를 여러 데이터 묶음으로 받아 각각의 화면 영역에 나누어 바인딩하는 방향을 정했다.
- 기본정보, 담당자 참관 여부, 파트 정보, 수지, 코팅, 전사 영역으로 상세 데이터를 분리하는 구조를 검토했다.
- 개정버전, 개정일, 개정상태, 배포일 정보를 기본정보 영역에 함께 표시하는 방식을 정리했다.
- 화면 초기화 시 바인딩 자체를 끊지 않고 기존 데이터셋을 비우는 방식이 더 안전하다는 점을 확인했다.
- 확정 버튼에서 현재 문서번호와 신규 여부 상태를 검증한 뒤 확정 처리만 호출하는 구조를 설계했다.
- 신규 작성, 최초 저장, 조회 선택, 확정 이후의 전역 상태값 관리 기준을 정리했다.

## 기술 스택 및 도구

- C# / WinForms
- SQL Server / T-SQL
- DataTable / DataSet
- 그리드 데이터 바인딩
- 문서 생명주기 기반 조회 및 확정 처리

## 문제 해결

- 상세 조회가 여러 결과 묶음을 반환하므로 단일 테이블 조회 방식으로는 화면 전체를 채울 수 없다는 점을 확인했다.
- 목록 그리드는 표시용 컬럼만 유지하고, 상세 조회에는 선택된 문서번호를 기준으로 접근하는 방식으로 정리했다.
- 화면 데이터 초기화 시 바인딩을 해제하면 그리드 상태가 꼬일 수 있어, 바인딩된 데이터 구조를 비우는 방식으로 방향을 잡았다.
- 확정 처리 전에 실제 문서번호가 확보되어야 하므로 신규 여부와 문서번호 전역 상태를 명확히 관리하도록 기준을 세웠다.

## 결과

- 목록 조회와 상세 조회의 역할이 분리됐다.
- 상세 조회 결과를 화면 기본정보와 여러 그리드로 나누어 바인딩하는 흐름이 정리됐다.
- 개정 관련 표시 정보의 바인딩 위치와 처리 방향이 정해졌다.
- 확정 처리 함수의 입력값, 검증 조건, 처리 후 화면 상태 전환 기준이 정리됐다.
